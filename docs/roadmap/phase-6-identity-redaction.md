# Phase 6 — Identity (DID/VC) + Redaction / Selective Disclosure

> Institutional identity and privacy-preserving disclosure/erasure that **never invalidates a proof**. **Source:** [architecture.md §9.1](../research/architecture.md) (identity), [§9.2](../research/architecture.md) (redaction), [§4.7](../research/architecture.md) (salted commitments), [§5.2](../research/architecture.md) (Participant/`vc_ref`), [§11.3](../research/architecture.md) (verify non-claims), [§18 ADRs](../research/architecture.md); [quality-foundations.md Domain 6](../research/quality-foundations.md) (crypto & key hygiene); [roadmap.md §6 Phase 3](../research/roadmap.md), [§4(c)](../research/roadmap.md), [FR5/FR7](../research/roadmap.md).

## Objective

Add the institutional-identity and privacy/erasure layers on top of the already-frozen Tier-1 seam. `did:key` is already pure-in-core from [Phase 2](phase-2-log-signing.md) (`core::did_key`, ADR-0010); this phase adds **`did:web` institutional resolution** with document-digest pinning, **AI-agent DIDs**, **Verifiable Credentials (VCDM 2.0)** bound by digest, and the three composable **redaction / selective-disclosure schemes** (salted-hash field redaction, redactable-Merkle subtree pruning, crypto-shredding). The non-negotiable property is **remainder verification**: `verify(entry) == verify(redact(entry, policy))` for every proof over non-redacted parts ([§9.2](../research/architecture.md)). All of this rests on the capture-time salted commitment ([§4.7](../research/architecture.md)) — the salt is external by construction (ADR-0012), so redaction is a pure transform over already-committed bytes that changes no digest a prior proof or anchor depends on. This satisfies [FR5](../research/roadmap.md) (resolve participant DIDs) and [FR7](../research/roadmap.md) (selective disclosure / redaction that preserves verifiability of the remainder, for GDPR erasure + medical confidentiality), and keeps invariant **I7** (integrity-not-validity-not-faithfulness) live *at the identity layer* via the honesty band.

## Deliverables

- **`thoughtmark-identity` crate — `IdentityResolver` trait** (arch [§9.1](../research/architecture.md)). The async resolver seam (`resolve(&Did) -> ResolvedDid`, `resolve_method(&DidUrl) -> VerificationMethod`); `Did`/`DidMethod{Key,Web}`/`DidUrl`/`VerificationMethod`/`PublicKeyMaterial::Ed25519([u8;32])` (the enum reserves room; Ed25519 only in v1). The pure `did:key` decoder already lives in `core::did_key` from [Phase 2](phase-2-log-signing.md) (ADR-0010) and is **not** re-implemented here — `thoughtmark-identity` is the std/I-O shell that wraps it for the `did:web` path.
- **`did:web` resolver — `DidWebResolver<F: DidWebFetcher, C: Clock>`** (arch [§9.1](../research/architecture.md)). Maps the DID to its HTTPS URL, fetches via the **injected** `DidWebFetcher` (no ambient network in core — **I3/I8**), JCS-canonicalizes the document through the one choke point (`serde_json_canonicalizer`, **I2**, ADR-0001) and records its **`document_digest`** so resolution is itself auditable. **Pins/snapshots the digest at first use and treats any later change as an explicit rotation event** (the #1 institutional-identity failure mode). Extracts Ed25519 `verificationMethod`s, accepting **both** `Multikey`/`publicKeyMultibase` **and** `publicKeyJwk` OKP. `Clock` is injected for `fetched_at`; **never** `SystemTime::now` (**I3**, clippy `disallowed-methods`, bootstrap row #6).
- **AI-agent DIDs** (arch [§9.1](../research/architecture.md), [§5.2](../research/architecture.md)). Either an ephemeral per-session `did:key` (orchestrator/TEE-held) or a stable institutional `did:web:org:agents:<model-id>`; either is a first-class `Participant{kind: Ai}` whose `DidUrl` is the DSSE `keyid`. The self-reported model string stays in `Participant.model_self_reported_version` (the field name encodes "reported, not third-party-attested").
- **Verifiable Credentials (VCDM 2.0) + `CredentialVerifier`** (arch [§9.1](../research/architecture.md)). VCs bind affiliation/role ("DID X is a licensed physician at Y") via the issuer's `did:web`; verified by `CredentialVerifier` over `ssi-vc`/`ssi-claims`, supporting Data Integrity **`eddsa-jcs-2022`** or **SD-JWT-VC**. The **full VC is held off-ledger** (it may carry PII; crypto-shreddable); only **`{vc_ref: Digest, issuer, types}`** is bound on-ledger (arch [§5.2](../research/architecture.md): `Participant.vc_ref: Option<Digest>`). `Participant.id` is a DID and **NEVER PII**.
- **The non-repudiable vs attested honesty band** (arch [§9.1](../research/architecture.md), surfaced in the [§15](../research/architecture.md) UI). *Non-repudiable* = "the private key behind DID X produced a `verify_strict`-valid Ed25519 signature over exactly these canonical bytes" (all the signing layer proves). *Attested* = "DID X belongs to Dr. Doe / is GPT-5 in a genuine TEE / the human reviewed this" (only as good as the VC issuer / TEE attestation / a UI assertion). The verifier visually separates **signature valid (cryptographic)** from **claims attested by issuer Z (trust-scoped)** — this is **I7** at the identity layer, and it dovetails with the permanent `NotEstablished.authorship_truth` non-claim in [§11.3](../research/architecture.md).
- **`thoughtmark-redaction` crate — `DisclosableLeaf`** (arch [§9.2](../research/architecture.md), [§4.7](../research/architecture.md)). The off-ledger disclosable-field representation: `Disclosed{salt, value}` / `Redacted{digest}` / `Shredded{ciphertext_digest, key_id}`. **All three yield the SAME on-ledger `ContentDigest::Hashed.digest_hex`**, so redaction is invisible to the Merkle/anchor proofs.
- **Scheme (1) — salted-hash field redaction (SD-JWT / RFC 9901)** (arch [§9.2](../research/architecture.md)). Per-field selective disclosure (reveal the AI rationale, redact a patient identifier inside it). Follows RFC 9901 exactly: disclosure array `[salt, name, value]`; `digest = base64url-nopad(SHA-256(ASCII(disclosure_string)))`; `_sd`/`{"...":digest}`; `_sd_alg: "sha-256"` — for SD-JWT-VC interop. This is a **separate SHA-256 hash domain** (RFC-9901 128-bit salts) from the Tier-0 BLAKE3 content path; the byte-identity rule is to fix one JSON serialization (our JCS, **I2**) and one salt→base64url routine and treat the produced **disclosure *string* as the immutable artifact** (RFC 9901 hashes the string, not re-derived JSON). *Preserves:* the credential signature, inclusion proof, later-disclose-a-subset, non-correlatability. *Destroys:* the plaintext value (but **not** the existence/shape leak of `{"...":digest}` — see Risks).
- **Scheme (2) — redactable-Merkle subtree pruning** (arch [§9.2](../research/architecture.md), [§6.1](../research/architecture.md)). Drop a whole turn/entry while keeping the rest provable: replace the pruned leaf with its known `TreeHash` (or keep only a subtree root). The **root is unchanged**, so all anchored roots and consistency proofs stay valid; surviving leaves' audit paths now include the pruned hash. RFC 6962 `0x00`/`0x01` leaf/node separation ([§6.1](../research/architecture.md)) prevents leaf-as-node confusion. *Preserves:* root, all proofs, and the count and position (**you can prove "turn #7 existed and was removed"**). *Destroys:* the pruned turn's content and the ability to later prove its *contents*.
- **Scheme (3) — crypto-shredding (GDPR Art. 17 / medical confidentiality)** (arch [§9.2](../research/architecture.md)). When even the salted digest must become meaningless: encrypt the plaintext at capture with **XChaCha20-Poly1305** (`chacha20poly1305`), a **24-byte injected nonce** (**I3** — never generated in core), a **per-subject key**; commit the *ciphertext's* salted hash; store keys in a deletable **`ShredKeyStore`**. Erasure = delete the key (EDPB Guidelines 02/2025). *Preserves:* ledger structure, root, all proofs, and the fact an entry existed at T. *Destroys (irrecoverably, for everyone):* the plaintext. This is the **only** scheme fit for medical PII and the only one that truly erases low-entropy values. Key material is wrapped in `secrecy::Secret`/zeroized on `Drop` (**I6**, bootstrap row #19).
- **`RedactionPolicy` / `RedactionScheme` / `RedactionReason` / `RedactedEntry`** (arch [§9.2](../research/architecture.md)). `RedactionPolicy{fields: Vec<FieldPath>, scheme, reason}`; `RedactionScheme{SaltedHash, PruneEntry, CryptoShred}`; `RedactionReason{GdprErasure, MedicalConfidentiality, LegalHold, Minimization}`; `RedactedEntry{entry, redaction_record}` — **the redaction record is itself an auditable ledger action** (a `RedactionRecord` appended to the ledger, not a silent mutation).
- **The pure `redact()` function** (arch [§9.2](../research/architecture.md)). `redact(&LedgerEntry, &RedactionPolicy, &RedactCtx) -> Result<RedactedEntry, Error>`. Operates **only on already-committed bytes**; **MUST NOT recompute any proof-bearing digest**; returns `Err` if a field was not committed disclosable at capture. Key deletion (shred) is a **separate, impure** `ShredKeyStore::shred` call — `redact()` stays pure (no I/O, no clock, no RNG) so it is in the byte-identity oracle set. `ShredKeyStore{get(&KeyId), shred(&KeyId)}`.
- **ADR-0011 — sign-over-the-salted-commitment only** (arch [§9.2](../research/architecture.md), [§18](../research/architecture.md)). v1 uses only the SD-JWT model: the signature covers the salted commitment, never the plaintext, so redaction never touches signed bytes and `verify_strict` still passes (keeps `ed25519-dalek`/`verify_strict` the sole signature primitive — **I6**). True redactable-signature schemes (homomorphic / Merkle RSS) are **deferred research** (unaudited relative to `ed25519-dalek`). Cross-checked against **ADR-0012** (salt is external to the hashed leaf — a salt committed inside signed/logged bytes could never be deleted; **I5**).
- **Reference-app wiring** (arch [§15.4](../research/architecture.md), [§15.5](../research/architecture.md)). Wire `vc_ref` plus the **off-ledger salt/VC store** and **crypto-shred key store** into the [Phase 5](phase-5-reference-app.md) MedQBank app's RLS-protected Supabase Storage: the on-ledger `ledger_entry`/`dsse_envelope` tables carry only salted hashes + DIDs (never content, never PII), while the disclosure salts, full VCs, and shred keys live in service-role-only storage buckets that can be deleted to effect erasure. The verifier UI renders the **green "signature valid (cryptographic)"** vs **amber "claims attested by issuer Z"** identity bands (arch [§15.5](../research/architecture.md)).

## Quality gates (must be green)

This phase newly **activates** (not the whole net again):

- **Remainder-verification oracle vector** (qf [Domain 4](../research/quality-foundations.md) conformance corpus, bootstrap row #10; arch [§9.2](../research/architecture.md), [§13](../research/architecture.md)) — `verify(entry) == verify(redact(entry, policy))` for **every** proof over non-redacted parts, byte-checked in `spec/vectors/redaction/**` across native Rust and WASM (**I1**). This is the load-bearing gate of the phase: each redaction step consumes digests that did not change.
- **Crypto-shred erasure tests** (qf [Domain 6](../research/quality-foundations.md), bootstrap row #19) — after `ShredKeyStore::shred`, the committed digest is **meaningless** (plaintext irrecoverable) while inclusion/consistency/anchor proofs **still pass** (`verify` `total = true` over the surviving structure). Key material is `zeroize`d on drop and never logged (secret-free error model, arch [§10.2](../research/architecture.md)).
- **SD-JWT / RFC 9901 disclosure vectors** (qf [Domain 6](../research/quality-foundations.md) authoritative vectors, bootstrap row #20) — authoritative RFC 9901 disclosure-string vectors (`base64url-nopad(SHA-256(ASCII(disclosure)))`, `_sd_alg: "sha-256"`) in `spec/vectors/redaction/sdjwt/**`; the disclosure *string* is treated as the immutable artifact.
- **New fuzz targets** (qf [Domain 2](../research/quality-foundations.md), bootstrap row #9) — a fuzz target per new parser: the `did:web` document parser and the SD-JWT/RFC-9901 disclosure parser (per the activation rule, "a fuzz target per new parser"). `redact()` is fuzzed for the invariant "never recomputes a proof-bearing digest / never panics."
- **VC verification + `did:web` resolution tests** — `eddsa-jcs-2022` Data Integrity and SD-JWT-VC verification via `ssi-vc`/`ssi-claims`; `document_digest` pinning test asserting a changed document is reported as a **rotation event**, not silently accepted.
- **Additive-only semver** (qf [Domain 2](../research/quality-foundations.md) `cargo-semver-checks`, [Domain 5](../research/quality-foundations.md), bootstrap row #5/§16) — new `RedactionScheme`/`RedactionReason` variants and any new `ApprovalScope` variant land additively against the [Phase 3](phase-3-verify-apifreeze.md) frozen `#[non_exhaustive]` enums; `cargo-semver-checks` blocks any accidental break of the 1.0 surface.
- **Key hygiene / secret scanning** (qf [Domain 6](../research/quality-foundations.md), bootstrap rows #19, #21) — `subtle`/`zeroize`/`secrecy` enforced on the new shred-key paths; `gitleaks` + push protection cover the new key-handling code so a committed key is impossible to push.

## Exit / freeze gate

A field can be **redacted or crypto-shredded and the trail still verifies** (remainder-verification oracle green; `verify` `total = true` over surviving structure with the shredded plaintext irrecoverable), and **`did:web` resolves with document-digest pinning** (a changed document surfaces as an explicit rotation event). No freeze is created or moved here: the **wire-format freeze** ([Phase 2](phase-2-log-signing.md)) and the **API/1.0 freeze** ([Phase 3](phase-3-verify-apifreeze.md)) both already hold; this phase's additions are additive (new optional crates + additive enum variants) and `core` stays **1.x**. Per [§16](../research/architecture.md) the format-identifier evolution rule governs: nothing here changes a hashed byte, so `canon_version` is untouched.

## Risks & open implementation-gaps

- **`did:web` key rotation is the #1 institutional-identity failure mode** (arch [§9.1](../research/architecture.md)). Mitigated by **document-digest pinning**: snapshot the `document_digest` at first use and treat any change as an explicit, auditable rotation event rather than a silent re-key. A naive resolver that always trusts the live document would let an attacker who controls the HTTPS host retroactively re-point a DID.
- **Low-entropy values defeat salted-hash redaction** (arch [§9.2](../research/architecture.md)). Salted-hash field redaction leaves an **existence/shape leak** (`{"...":digest}` reveals a field existed; a low-entropy value like a yes/no or a small enum can be brute-forced through the salted commitment). Such fields **must** use crypto-shred (scheme 3), the only scheme that truly erases low-entropy values — composing this correctly is per-field policy work.
- **Composing the right scheme per field is design work** (roadmap [§4(c)](../research/roadmap.md), arch [§9.2](../research/architecture.md)). Selective disclosure is only PARTIALLY-SOLVED at the primitive level; choosing salted-hash vs prune vs crypto-shred for each conversation field (AI rationale, patient identifier, system prompt, tool args) is genuine design effort and a place for mistakes — captured as a `RedactionPolicy` that should be reviewed, not improvised.
- **No exotic redactable-signature crypto in v1** (ADR-0011, arch [§9.2](../research/architecture.md)). True redactable-signature schemes (homomorphic / Merkle RSS) are deferred as research because they are unaudited relative to `ed25519-dalek` (**I6**); v1 deliberately accepts the sign-over-salted-commitment model's limits rather than introduce unaudited signature math.
- **Salt/key custody is the real erasure boundary** (arch [§4.7](../research/architecture.md), ADR-0012). Erasure is only as good as deletion of the off-ledger salt/ciphertext/shred-key; the on-ledger commitment is permanent by design. The reference app must ensure the salt/VC/shred-key stores are genuinely deletable (service-role-only, no orphaned backups) for GDPR Art. 17 / medical-confidentiality claims to hold.
- **SD-JWT byte-identity discipline** (arch [§9.2](../research/architecture.md)). RFC 9901 hashes the disclosure *string*, not re-derived JSON; the implementation must fix exactly one serialization + one salt→base64url routine and treat the string as the immutable artifact, or the SD-JWT digests will diverge from the SD-JWT-VC ecosystem and from the WASM build (**I1/I2**).
