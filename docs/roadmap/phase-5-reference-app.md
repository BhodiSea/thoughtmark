# Phase 5 — Reference App (MedQBank, Next.js/Supabase)

> A *thin* demonstrator that proves the library end-to-end — a demo of the library, never the product, and never at the cost of the trust story. **Source:** [roadmap §6 Phase 2](../research/roadmap.md), [architecture §15](../research/architecture.md) (all subsections), [architecture §12.2](../research/architecture.md), [architecture §9.1](../research/architecture.md), [quality-foundations Domain 3](../research/quality-foundations.md).

## Objective

Stand up `apps/medqbank` — a Next.js (App Router, **Node runtime**) + Supabase (Postgres/Auth/Storage) study platform — whose single job is to drive the published 1.0 `@thoughtmark/core` through one complete lifecycle: a student study session is **captured** as an append-only `ContributionLedger`, its per-session Merkle root is **anchored** via Tier 2 (Phase 4), and an uploaded `ThoughtmarkBundle` is **verified** statelessly by the same `core.verify` the library ships ([architecture §15](../research/architecture.md)). The app exists to demonstrate that the trust story holds in a realistic integration — so the demonstrator must *itself* embody the invariants (server-only WASM, salted-hashes-only, RLS-everywhere) rather than work around them. It must not introduce a second WASM build, a client-side key, or a content-bearing ledger table. This is exactly [roadmap §6 Phase 2](../research/roadmap.md): "student question → AI explanation turns → student refinement → final answer," producing a downloadable proof bundle + a C2PA export of the final artifact.

## Deliverables

- **`apps/medqbank` — WASM runs server-side only, Node runtime** ([architecture §15.1](../research/architecture.md), [architecture §12.2](../research/architecture.md)). The app consumes the *single* `@thoughtmark/core` `--target web` artifact via the package's `node` entry — **not** a second `--target nodejs` build (one artifact = one byte-audited binary, per [phase-2-log-signing.md](phase-2-log-signing.md) and [architecture §12.2](../research/architecture.md)). Encode the four Node gotchas:
  - `import 'server-only'` atop `lib/core/index.ts` (a stray client import becomes a build error → the signing/canon path can never reach the browser).
  - `export const runtime = 'nodejs'` on **every** route/page that transitively touches core (Edge/workerd lacks `fs` and instantiates WASM differently — the determinism-injection and key-custody arguments both fail there).
  - `serverExternalPackages: ['@thoughtmark/core']` in `next.config.ts`.
  - a module-level `loadCore()` singleton so the wasm instantiates exactly once per Node process.
  Rationale (all from [architecture §15.1](../research/architecture.md)): **key custody** (sealing needs the server-custodied Ed25519 key, so signing is a Server Action — never a browser secret/oracle), **single byte-identity oracle** (one runtime produces canonical bytes for both capture and verify, removing "browser WASM produced subtly different bytes" from the demo's trust surface — the browser artifact still exists and is exercised only by the [§13 byte-parity test](../research/architecture.md) wired in [phase-2-log-signing.md](phase-2-log-signing.md)), and **determinism injection** (the Server Action is the natural place to read the capture `Clock` and CSPRNG salt and pass them into pure core, honoring [I3](../research/architecture.md)).

- **Capture flow — a study session is an append-only `ContributionLedger`** ([architecture §15.2](../research/architecture.md)). State machine `OPEN → FINALIZED → ANCHORED`. The **`appendTurn` Server Action**, atomically server-side:
  1. **injects** `attested_at` from the server `Clock` (→ `UnixMillis`) and a **32-byte per-content salt** from the server CSPRNG ([I3](../research/architecture.md), [I5](../research/architecture.md));
  2. computes each `ContentDigest::Hashed` via `salted_content_digest`, with the salt stored **off-ledger** in RLS-scoped Supabase Storage — **never** inside the canonical `Turn` (salt-external-to-the-leaf, [ADR-0012](../research/architecture.md), so it stays crypto-shreddable);
  3. assembles the multi-part `Turn` and its `ContributionLedger` entry; builds the in-toto `Statement` (predicate = the `Trail` prefix), JCS-canonicalizes through the one choke point ([I2](../research/architecture.md)), and **DSSE-signs** with the server key (exactly one signature per sealed turn, [ADR-0007](../research/architecture.md); `ed25519-dalek` only, [I6](../research/architecture.md));
  4. `append`s to the per-session Merkle log via `thoughtmark-log`'s `PostgresStorage`;
  5. inserts `ledger_entry` + `dsse_envelope` and updates `session_log_state` in **one service-role transaction** with an **optimistic guard on `size`** so concurrent turns cannot fork the log.
  **`finalizeSession`** appends a terminal `answer` turn and stores the **final canonical artifact bytes** in Storage for the C2PA export. **Raw turn bodies are never in the ledger tables** — only salted commitments + metadata; bodies live in RLS-scoped Storage, crypto-shreddable (matures in [phase-6-identity-redaction.md](phase-6-identity-redaction.md), [architecture §9.2](../research/architecture.md)).

- **Periodic root anchoring job — two-level Merkle** ([architecture §15.3](../research/architecture.md)). Each session has its own append-only log; a periodic job builds a **batch tree over the session roots changed since the last anchor** and anchors only that single batch root (Tier 2, from [phase-4-tier2-anchoring.md](phase-4-tier2-anchoring.md)). Mechanism: `pg_cron` (`*/10 * * * *`) → `pg_net.http_post` → an `anchor-roots` **Edge Function that is a thin scheduler shim** (secrets via Supabase **Vault**) which POSTs back to the Node `/api/anchor` route — **all core crypto stays in the one Node runtime** (the single-oracle principle from [architecture §15.1](../research/architecture.md); the Edge function holds no core logic). The route calls the `thoughtmark-anchor-ots` plugin (calendar `submit` → pending `.ots`, [ADR-0008](../research/architecture.md)/[ADR-0009](../research/architecture.md)) and writes `anchor_receipt` + `batch_inclusion` rows. A **second hourly cron** sweeps `pending → confirmed` via OTS `upgrade` once the Bitcoin attestation lands.

- **Postgres schema + RLS** ([architecture §15.4](../research/architecture.md)). Eight tables — `study_session`, `ledger_entry`, `dsse_envelope`, `merkle_nodes`, `session_log_state`, `checkpoint`, `anchor_receipt`, `batch_inclusion` — with the DDL sketch in [architecture §15.4](../research/architecture.md). Principles, enforced exactly:
  - **RLS on every table.**
  - **All writes go through service-role server code** — *no* client write policies on ledger/log/anchor tables (so clients cannot write the ledger at all).
  - Students get **read-only** policies scoped to their own sessions (`sess_read` / `ledger_read` keyed on `auth.uid() = owner`); `anchor_receipt` is public-read (no sensitive content), service-role write only.
  - Only salted hashes + metadata are stored — **never content** ([I5](../research/architecture.md)). `ledger_entry.leaf_hash` is the 32-byte SHA-256 `TreeHash` of the DSSE leaf; `dsse_envelope.payload_b64` is base64 of the canonical `Statement`.
  - **Append-only enforced doubly**: no client write policies **and** `REVOKE UPDATE/DELETE ON ledger_entry/dsse_envelope FROM service_role` + a reject-mutation trigger.
  - **Two Supabase clients**: `createServerClient()` (anon key + user JWT, RLS-bound, reads only) vs `createServiceClient()` (service-role key, **no user JWT attached** so it genuinely bypasses RLS for ledger writes).

- **Verifier UI `/verify/[id]` (Node runtime)** ([architecture §15.5](../research/architecture.md), [architecture §9.1](../research/architecture.md)). Re-runs the **same** `core.verify(bundle, policy, now, anchorVerifier)` over an uploaded `ThoughtmarkBundle` ([architecture §7.6](../research/architecture.md)) — **stateless, no DB access, fully offline** (the verify path published in [phase-3-verify-apifreeze.md](phase-3-verify-apifreeze.md)). Honest copy: "This trail **EXISTED on or before** `<anchor/TST time>`, with this human/AI lineage, and has **NOT been altered since capture**," reporting `existed_at_or_before` + `unaltered_since_capture`, plus the explicit `NotEstablished` disclaimers (`validity_of_record`, `faithfulness`, `authorship_truth`, `completeness`) from [architecture §11](../research/architecture.md) — never validity, never faithfulness ([I7](../research/architecture.md)). Renders identity in **two visually separate bands** ([architecture §9.1](../research/architecture.md)): a **green** "signature valid (cryptographic)" band (the `verify_strict`-valid Ed25519 signature over exactly these canonical bytes) vs an **amber** "claims attested by issuer Z" band (the VC/TEE/UI assertion that the DID belongs to Dr. Doe / is model M). Reusable components may optionally be extracted as `@thoughtmark/verifier-ui`.

- **`exportC2PA` — external/sidecar manifest, artifact-level only, produced by an external C2PA SDK in app code** ([architecture §15.5](../research/architecture.md)). At this phase the sidecar is produced **directly by the upstream C2PA reference implementation (`c2pa-rs`, with its CAWG Identity Assertion support) invoked from the app's Node Server Action** — **not** by a `thoughtmark`-owned crate. The `thoughtmark-c2pa`/CAWG adapter plugin and the `core.export(ProvO|C2pa)` verb **do not exist yet** (they land in [phase-8-interop-standardization.md](phase-8-interop-standardization.md), [architecture §17 Phase 5](../research/architecture.md)); so this `exportC2PA` is an app-level helper that calls the external C2PA SDK over the finalized artifact bytes already stored by `finalizeSession`, never the (unbuilt) thoughtmark export surface. It emits a C2PA **external/sidecar** manifest with a **DataHash hard binding** over the final-artifact bytes + a **CAWG Identity Assertion** (human DID/VC + AI model id) + a plain-text Content Credentials summary, shipped **alongside the `.tmbundle`**. This is artifact-level only. (Per [roadmap §6](../research/roadmap.md): C2PA "text support" is the v2.3+/Jan-2026 steganographic blob embedding, not a turn schema — treat as an export target, not the core model.)

## Quality gates (must be green)

This phase **activates** the Next.js/Supabase data-layer gates from [quality-foundations Domain 3](../research/quality-foundations.md) (the Next.js/Supabase baseline; bootstrap-row TS gates **[#12–14](../research/quality-foundations.md)** apply to the app's TS surface). These are *new* to the gate net because the app code they target first appears here:

- **`supabase gen types typescript` checked into `database.types.ts`, with drift failing the build** ([architecture §15.5](../research/architecture.md), [quality-foundations Domain 3](../research/quality-foundations.md)) — a generated-vs-committed diff is a required CI step so the TS types can never silently diverge from the live schema.
- **pgTAP RLS test** ([architecture §15.5](../research/architecture.md)) — proves an `authenticated` client **cannot insert into `ledger_entry`** *and* **cannot read another user's session**. This is the machine-checkable proof that the data layer does not undercut the trust story.
- **Next.js ESLint (flat config, `eslint-config-next`), run directly** ([quality-foundations Domain 3](../research/quality-foundations.md)) — `next lint` is **removed in Next.js 16** and the `eslint` key in next config is no longer needed; confirm the app's Next major and wire ESLint directly. Core `@thoughtmark/core` lint/format remains Biome (Domain 3); ESLint is scoped to the app only.
- **RLS on every table + Supabase advisors run in CI** ([quality-foundations Domain 3](../research/quality-foundations.md)) — the advisors catch missing RLS / exposed policies before merge.
- **Service-role keys server-side only** ([quality-foundations Domain 3](../research/quality-foundations.md)) — enforced by `server-only`, secret scanning (bootstrap row [#21](../research/quality-foundations.md)), and Vault for the Edge shim; the service-role key never reaches a client bundle or an Edge function body.

(All [I1–I8](../research/architecture.md) gates already standing from prior phases — byte-parity [§13](../research/architecture.md), `forbid(unsafe)`, no-float, audited-crypto — remain authoritative; this phase adds no new core-crate gate because no new *core* parser lands here.)

## Exit / freeze gate

A study session is **captured → anchored → verified end-to-end via the published 1.0 API** — i.e. `appendTurn`/`finalizeSession` seal turns into the `ContributionLedger`, the two-level Merkle batch root is anchored through `thoughtmark-anchor-ots`, and `/verify/[id]` returns a green `unaltered_since_capture` + a bounded `existed_at_or_before` over the resulting `ThoughtmarkBundle`. The artifacts of a session are downloadable: a **`.tmbundle`** + its **C2PA sidecar manifest**. **No freeze is created or moved here** — Phase 5 consumes the wire format frozen at the end of [phase-2-log-signing.md](phase-2-log-signing.md) and the API/1.0 freeze published at the end of [phase-3-verify-apifreeze.md](phase-3-verify-apifreeze.md); the core crate stays 1.x and the demonstrator must not require any change to it ([architecture §17 Phase 5](../research/architecture.md)).

## Risks & open implementation-gaps

- **A forwarded user JWT re-enabling RLS on the service client** ([architecture §15.4](../research/architecture.md)). `createServiceClient()` must attach the service-role key with **no user JWT**; if a user JWT is forwarded onto it, RLS silently re-engages and the ledger writes (which have *no* client write policy) start failing — or worse, partially succeed under the wrong identity. Pin this with the pgTAP test and a code-review invariant that the service client is never constructed from a request-scoped auth header.
- **Edge/workerd lacks `fs` and instantiates WASM differently** ([architecture §15.1](../research/architecture.md)) — hence WASM is **Node-only** and the `anchor-roots` Edge Function is a thin scheduler shim with zero core logic. Any drift toward running core in the Edge runtime breaks the single-oracle property; `export const runtime = 'nodejs'` + `server-only` are the structural guards.
- **Browser-side key custody** ([architecture §15.1](../research/architecture.md)) — signing is a Server Action precisely because the server-custodied Ed25519 key (or an equivalent signing oracle) must never reach the browser. A regression that moves sealing client-side is a trust-story failure, not merely a refactor.
- **OpenTimestamps has no turn-key Rust library** ([quality-foundations Appendix A.9](../research/quality-foundations.md), [architecture ADR-0009](../research/architecture.md)) — `rust-opentimestamps` 0.7.2 parses/replays only; the OTS calendar HTTP protocol is implemented natively in the plugin from [phase-4-tier2-anchoring.md](phase-4-tier2-anchoring.md). The `pending → confirmed` upgrade has Bitcoin-confirmation latency, so the demonstrator's anchored state is asynchronous by design (the hourly sweep), not instantaneous.
- **C2PA versioning moves fast** ([roadmap §6 Caveats](../research/roadmap.md)) — v2.3 text support landed Jan 2026, v2.4 followed; the sidecar here is deliberately **artifact-level only** (DataHash hard binding + CAWG Identity Assertion over the final artifact). The full conversation-aware adapter is deferred to [phase-8-interop-standardization.md](phase-8-interop-standardization.md), where the export must never re-hash (P6).
- **`supabase gen types` / Supabase platform drift** — the generated-types CI gate is only as good as the pinned Supabase CLI; treat a CLI bump as a reviewed change since it can alter `database.types.ts` shape independently of the schema.
